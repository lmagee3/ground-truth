## OBJECTIVE
Build and own the **verification pipeline** for Ground Truth — the quality layer that ensures every context report the engine produces is factually accurate, source-verifiable, and free from editorial bias. You are the last gate before anything goes public.

## WHY THIS MATTERS
Ground Truth's entire value proposition is **trust**. We claim to be the anti-propaganda, no-spin intelligence briefing. If we ship a single report with bad dates, broken source links, or undetected bias, our credibility is done. Your pipeline is what makes this product real.

## YOUR DELIVERABLES

### Phase 1 — Verification Framework (Week 2-3)

#### 1. Source Validator
**Location:** `groundtruth/verification/source_validator.py`

Build a module that takes a context report and validates every source citation:
- **URL check** — Does the source URL resolve? Is it still live?
- **Domain trust** — Is the domain on our approved authoritative source list? (See `docs/APPROVED_SOURCES.md`)
- **Claim-source alignment** — Does the cited source actually support the claim being made? (AI-assisted check)
- **Date freshness** — Flag sources older than expected for the context (e.g., citing 2015 data for a 2026 event without noting the age)
- **Archive fallback** — If a URL is dead, check Internet Archive Wayback Machine for cached version

**Output:** Validation report with pass/warn/fail for each source.

#### 2. Bias Detector
**Location:** `groundtruth/verification/bias_detector.py`

Build a module that scans context reports for editorial bias:
- **Language analysis** — Flag loaded/emotional language (e.g., "brutal regime" vs "government forces")
- **Perspective balance** — Does the report present multiple interpretive frameworks? Flag if only one narrative is presented
- **Attribution check** — Are claims attributed to specific sources, or presented as universal truth?
- **Framing analysis** — Check if the ordering/emphasis of events implies a particular narrative
- **Country/actor balance** — Flag if one side of a conflict gets significantly more detail or more favorable framing

**Output:** Bias score (0-100) with specific flagged passages and recommended rewrites.

#### 3. Fact Checker
**Location:** `groundtruth/verification/fact_checker.py`

Build a module that cross-references key claims:
- **Date verification** — Cross-check dates against multiple authoritative sources (GDELT, ACLED, UCDP)
- **Name/title verification** — Are leaders, organizations, treaties named correctly with current titles?
- **Statistics verification** — Cross-check numbers (casualties, economic figures, vote counts) against primary data
- **Event sequence** — Verify chronological ordering of timeline events
- **Geographic accuracy** — Verify location names, borders, jurisdictions

**Output:** Fact-check report with verified/unverified/disputed status for each key claim.

### Phase 2 — Integration Testing (Week 3)

#### 4. QA Test Suite
**Location:** `tests/verification/`

Build comprehensive tests for the verification pipeline:
- **Known-good reports** — Feed in manually verified reports, confirm pipeline passes them
- **Known-bad reports** — Feed in reports with intentional errors (wrong dates, dead links, biased language), confirm pipeline catches them
- **Edge cases** — Multiple valid dates for same event (different time zones), contested territories, disputed statistics
- **Regression tests** — As we fix false positives/negatives, add test cases to prevent regression

#### 5. Verification Report Template
**Location:** `docs/templates/verification_report.md`

Create a standard template for verification reports that includes:
- Overall confidence score
- Source validation summary (X of Y sources verified)
- Bias assessment summary
- Fact-check results
- Flagged items requiring human review
- Recommended improvements

### Phase 3 — Continuous QA (Week 4+)

#### 6. Pre-publish Gate
Every context report generated by the engine must pass through your pipeline before it's returned via the API. This means:
- Verification runs as part of the API request pipeline (async, cached)
- Reports that fail verification get flagged but still returned with a `verification_status` field
- Critical failures (broken sources, factual errors) trigger alerts
- Monthly QA reports on pipeline accuracy

## CONSTRAINTS & GUARDRAILS

1. **You review, you don't rewrite.** Flag issues and recommend fixes. The synthesis engine handles rewrites.
2. **False positives > false negatives.** Better to flag something clean than miss something bad. We can tune later.
3. **Speed matters but accuracy matters more.** Target <5 second verification per report, but don't sacrifice quality.
4. **No code commits without tests.** Every module needs corresponding test coverage.
5. **Report findings to Opus for triage.** If you find systematic issues (e.g., GDELT dates are consistently off by a day), escalate — don't try to fix the ingestion layer yourself.
